The catat handbook.

A short, opinionated tour of how to ship verifiable forms on Sui + Walrus + Seal — written like notes, not specs.

01Welcome

catat is three things stacked: a builder that lays out fields, a runner that signs and submits replies, and an admin that decrypts what only you can read. It runs in the browser. There is no server. We host the UI on Walrus Sites; Sui keeps form state; Walrus stores blobs; Seal handles per-field keys.

If you can use Typeform, you already know catat. If you can read a Move module, you already know its guts. Read on for the parts that aren’t obvious.

📖
Reading order Skim Quickstart, then jump to whatever section matches the question you have. Specs live at the back; we won’t be offended.

02Quickstart (90s)

Connect a Sui wallet on testnet, fork a template, hit publish. Three commands also work:

# 1. install the cli
$ npm i -g @catat/cli

# 2. mint a form from a template
$ catat new bug-report --network testnet
→ form 0x4a2c…b91 minted on sui
→ schema bafy…2x9 written to walrus

# 3. share the link
$ catat link 0x4a2c…b91
→ https://catat.wal.app/f/0x4a2c…b91

That’s the whole loop. From here, every reply emits a SubmissionAdded event and stores a Quilt-batched Walrus blob.

03Data model

A Form is a Sui shared object pointing at a schema blob on Walrus. Each Submission is a separate event tied to that form, with its own Walrus pointer. Sealed fields hold ciphertext + a SealKeyId; everything else is plaintext JSON.

fieldtyperequirednote
idObjectID*shared object on sui
schemaBlobId*walrus pointer to the form spec
owneraddress*only this address can decrypt sealed fields
token_gateOption<ObjectID>—require holding this coin/NFT
epochsu32—walrus retention, default 12
countu64—monotonic submission counter

04Sealed fields

Toggle 🔒 on any field to encrypt its body before it leaves the browser. catat derives a per-field Seal key id of the form formId/fieldId, encrypts client-side, and stores ciphertext in the Walrus blob. Decryption requires a 2-of-3 threshold against your wallet.

🔒
Important Once a field is sealed, you cannot change the policy retroactively. Past submissions stay sealed under the original key id.

Encrypt a field with the SDK directly:

import { sealField } from "@catat/sdk";

const ct = await sealField({
  formId: "0x4a2c…b91",
  fieldId: "email",
  plaintext: "hex@catat.io",
  threshold: "2-of-3",
});

05Walrus & Quilt

One submission can have many fields, including files. Naively, that would be one Walrus write per file. Instead we batch the whole submission into a single Quilt — one atomic write, one blob id, one tx event. The math:

shapewritescost (testnet)relative
5 fields, 2 attachments — naive7~0.018 SUI1×
5 fields, 2 attachments — quilt1~0.00021 SUI~85× cheaper

06Token gating

Set token_gate on a form to require holding a specific coin or NFT. The form runner checks the wallet’s holdings on Sui state before allowing submit. Allowlists work the same way — each address holding a 1-of-N InvitePass NFT.

💡
Tip Combine gating with once_per_wallet=true for proposal votes: 1 wallet, 1 vote, no off-chain plumbing.

07Webhooks

On the Notebook plan, configure a webhook URL per form. We’ll POST a signed payload on every submit:

{
  "event": "submission.added",
  "form_id": "0x4a2c…b91",
  "tx_hash": "0xeb1d…c2f",
  "blob_id": "bafy…h1k",
  "submitter": "0x9f1a…e2c",
  "sealed_fields": ["email", "screenshot"],
  "signature": "ed25519:…"
}

08Self-hosting

The whole frontend is a static bundle. Fork the repo, point VITE_PACKAGE_ID at your deployed Move package, and deploy to Walrus Sites with one command. No backend. No database. Your fork — forever.

09CLI reference

commanddoes
catat new <template>fork a template into a new form
catat publish <form>upload schema to walrus, mint sui object
catat seal <form.field>flip a field to sealed
catat list <form>list submissions (sealed bodies stay sealed)
catat decrypt <sub>decrypt sealed fields with your wallet
catat export <form>dump everything to ./out.json
